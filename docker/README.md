# Testing OrioleDB Docker images

The `docker` directory contains the Docker image definitions, initialization
scripts, and test hooks used to build and validate OrioleDB PostgreSQL images.
It supports both routine local build/test work and the repository CI workflows
that build, publish, and validate container images.

For normal local work, the primary entry point is `./ci/docker_matrix.sh` from
the repository root. Use `./ci/docker_matrix.sh --help` for the supported
build/test matrix, examples, side effects, and cleanup notes. The file
structure below provides the quickest orientation for the source layout, while
the manual official-images flow documented later in this file is the
lower-level path intended for isolated debugging of a specific prebuilt image
or test case.

Running the Docker Official Image tests against OrioleDB images, see:
* https://github.com/docker-library/official-images/tree/master/test

## File structure

Directory snapshot:

```text
.
├── Dockerfile
├── Dockerfile.ubuntu
├── README.md
├── config.guess
├── config.sub
├── init
│   ├── default-orioledb.sh
│   ├── docker-entrypoint.sh
│   └── postgresql.docker.conf
├── orioledb-config.sh
└── tests
    └── orioledb-basics
        └── run.sh
```

The main files and directories are described below.

- `Dockerfile` -- main image definition.
- `Dockerfile.ubuntu` -- Ubuntu-based variant used for matrix builds and CI
  coverage.
- `README.md` -- documentation entry point for the Docker-related workflow.
- `config.guess` and `config.sub` -- platform-detection helpers used during
  build/configure stages.
- `init/` -- container startup logic and PostgreSQL runtime defaults.
- `init/default-orioledb.sh` -- OrioleDB-specific default setup.
- `init/docker-entrypoint.sh` -- main image entrypoint implementation.
- `init/postgresql.docker.conf` -- PostgreSQL configuration tuned for the
  container image.
- `orioledb-config.sh` -- integration with the Docker Official Images test
  suite.
- `tests/orioledb-basics/run.sh` -- OrioleDB-specific smoke test used in the
  Docker test flow.

## Local development flow

For routine local work, `./ci/docker_matrix.sh` is the primary entry point.
It builds the Docker matrix across supported base images and build variants and
can also run the official image tests as part of the same workflow.

Use the manual official-images commands below only when low-level debugging is
required for one already-built image or one specific test.

## Example: Docker build parameterization

The Dockerfiles support a build-argument driven workflow. In practice, the
recommended way to inspect the effective `docker build` invocation is to let
`./ci/docker_matrix.sh` generate it.

The following example was generated by:

```bash
./ci/docker_matrix.sh --base ubuntu:24.04 --dry-run
```

Example build command:

```bash
docker build \
  --pull \
  --network=host \
  --progress=plain \
  --build-arg BASE_IMAGE="ubuntu" \
  --build-arg BASE_VERSION="24.04" \
  --build-arg BUILD_CC_COMPILER="clang" \
  --build-arg PG_MAJOR="17" \
  --build-arg DEBUG_MODE="false" \
  --build-arg PG_LLVM_ENABLED="true" \
  --label "org.opencontainers.image.title=OrioleDB 17-clang-ubuntu-24.04" \
  --label "org.opencontainers.image.description=OrioleDB: next-generation storage engine for PostgreSQL" \
  --label "org.opencontainers.image.url=https://www.orioledb.com/" \
  --label "org.opencontainers.image.licenses=Apache-2.0 OR PostgreSQL" \
  --label "org.opencontainers.image.revision=beta13-202-g0220a176-2026-03-20-dirty" \
  --label "org.opencontainers.image.source=https://github.com/orioledb/orioledb" \
  -f "./docker/Dockerfile.ubuntu" \
  -t "orioletest:17-clang-ubuntu-24.04" .
```

This example shows the main customization axes:

* base distribution and version via `BASE_IMAGE` and `BASE_VERSION`
* PostgreSQL major version via `PG_MAJOR`
* compiler selection via `BUILD_CC_COMPILER`
* debug/release behavior via `DEBUG_MODE`
* LLVM-enabled PostgreSQL build path via `PG_LLVM_ENABLED`
* image identity via the final tag and OCI labels

The resulting image can then be tested with the Docker Official Images test
suite. In this workflow, `./log_docker_build/official-images/` is a local
checkout of `docker-library/official-images`.

Example test command:

```bash
./log_docker_build/official-images/test/run.sh \
  -c "./log_docker_build/official-images/test/config.sh" \
  -c "docker/orioledb-config.sh" \
  "orioletest:17-clang-ubuntu-24.04"
```

For most local and CI-oriented work, prefer using `./ci/docker_matrix.sh`
rather than composing these `docker build` arguments by hand.

If you are changing `./docker/Dockerfile` or `./docker/Dockerfile.ubuntu`,
also check whether the change affects:
* `./.github/workflows/dockertest.yml`
* `./.github/workflows/docker.yml`
* `./ci/docker_matrix.sh` arguments, help text, examples, tag naming, log
  handling, or expected test behavior
* `./doc/contributing/docker-builds.mdx`

## Support priority

Support priority is not equal for every Dockerfile customization path.

Tier 1: highest priority
* Correct behavior of the Dockerfiles with the `--build-arg` combinations
  actually used by:
  * `./.github/workflows/docker.yml`
  * `./.github/workflows/dockertest.yml`
* In practice this means the default CI/publishing paths are primary, for
  example combinations around:
  * `PG_MAJOR`
  * `ALPINE_VERSION`
  * `UBUNTU_VERSION`
  * `BUILD_CC_COMPILER`
  * `DOCKER_PG_LLVM_DEPS=llvm-dev clang`
  * `DEBUG_MODE=false`

Tier 2: lower priority
* Other custom `--build-arg` combinations that are not used by the GitHub
  workflows, especially non-default/custom exploratory combinations.
* Local `./ci/docker_matrix.sh` convenience behavior, help text, and matrix
  coverage outside the main CI/publishing paths.

Interpretation:
* If a change breaks a Tier 1 workflow path, fix that first.
* If a problem affects only Tier 2 matrix/customization paths, it is still
  worth documenting and may still be fixed, but it is not the first priority.

## Running docker test suite

This section is the low-level/manual official-images test path.
If you need the integrated local matrix workflow, use:

```bash
./ci/docker_matrix.sh --help
```

```bash
# clone official-images test suite
OFFIMG_LOCAL_CLONE=./log_docker_build/official-images
OFFIMG_REPO_URL=https://github.com/docker-library/official-images.git
mkdir -p "$OFFIMG_LOCAL_CLONE"
git clone "$OFFIMG_REPO_URL" "$OFFIMG_LOCAL_CLONE"

"${OFFIMG_LOCAL_CLONE}/test/run.sh" \
    -c "${OFFIMG_LOCAL_CLONE}/test/config.sh" \
    -c "docker/orioledb-config.sh" \
    "orioletest:17-gcc-ubuntu-24.04"
```

If the test is ok, you can see:

```bash

	'utc' [1/6]...passed
	'no-hard-coded-passwords' [2/6]...passed
	'override-cmd' [3/6]...passed
	'postgres-basics' [4/6]....passed
	'postgres-initdb' [5/6]....passed
	'orioledb-basics' [6/6]...passed
```

## AI agent notes

For AI agents:
* Prefer `./ci/docker_matrix.sh --help` as the operational source of truth for
  arguments, image tag naming, log paths, marker files, and cleanup commands.
* After a matrix run, verify:
  * the per-run log directory under `./log_docker_build/<timestamp>-pid-<pid>`
  * `<docker_tag>.build.ok` and `<docker_tag>.test.ok` or `.test.skipped`
  * existence of `orioletest:<docker_tag>`
  * existence/removal of helper images such as `librarytest/postgres-initdb:*`
* When analyzing failures, separate:
  * repo-relevant breakage
  * upstream PostgreSQL or distro breakage
  * expected instability on development distro tags
* For machine-readable image checks, prefer:
  * `docker images --format '{{.Repository}}:{{.Tag}}' 'orioletest*'`
  * `docker images --format '{{.Repository}}:{{.Tag}}' 'librarytest/*:*'`
